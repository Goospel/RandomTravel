# RandomTravel — 설계 문서

> 최종 갱신: 2026-07-10

---

## 1. 개요

랜덤으로 국내 여행지를 추천하는 **반응형 웹앱**.

- **목적**: 가볍게, 본인이 직접 쓰는 재미 프로젝트. 과한 인프라·운영 부담 없음.
- **핵심 가치**: "어디 갈지 고민하는 시간" 자체를 없앤다. 결정은 주사위에 맡긴다.
- **부수 가치(분산)**: 시·도 균등 랜덤(§5.5)은 유명 관광지 쏠림을 깨는 **분산 장치**이기도 하다 — 이 서사를 UI 문구로 드러내는 원칙·카피는 §7.9(M18).
- **설계 원칙**: DB 없음 · 로그인 없음 · 별도 백엔드 서버 없음. 최소 구성으로 실제 동작하는 것.
  - 이 원칙은 **P0(MVP) 기준** — 데이터 수익화 단계(P1+)에선 이벤트용 **DB만 추가**된다. **로그인은 그때도 불필요**(통계는 익명 세션 ID로 충분, §12.5).

## 2. 핵심 경험 — 조건은 전부 선택사항, 비면 완전 랜덤

> **핵심 불변식**: 모든 뽑기는 같은 파이프라인을 탄다 — `지역 풀 → 필터 적용 → 지역·타입 확정 → 랜덤 추출(§5.5)`.
> **조건(필터)을 하나도 넣지 않으면 필터 0개 = 완전 랜덤.** 조건은 어디까지나 풀을 좁히는 장치다.
> **기준일(📅, §6.8)은 조건이 아니라 판정 시점 입력** — 날짜 단독 선택은 완전 랜덤과 동일하다(풀을 좁히지 않는다).

### 2.1 두 가지 모드 (UI 진입점)

| 모드 | 설명 | 흐름 |
|---|---|---|
| 🎰 **순수 랜덤** | 조건 0개로 즉시 뽑기. "운명에 맡긴다" 컨셉. | 메인의 큰 버튼 → 바로 결과 |
| 🎯 **조건 랜덤** | 아래 조건을 걸고, 그 범위 안에서 랜덤 | 조건 선택 → 뽑기 → 결과 |

두 모드는 메인 화면 상단 토글로 전환. 조건 모드에서 아무 조건도 켜지 않고 뽑으면 순수 랜덤과 완전히 동일하게 동작한다(같은 파이프라인이므로 자동 보장).

### 2.2 조건(필터) 체계

| 분류 | 조건 | 동작 | 데이터 소스 |
|---|---|---|---|
| 정적 | 지역 (17개 시·도, 다중) | 지역 풀을 선택 지역으로 한정 | 상수 |
| 정적 | 테마 (관광 타입 8종, 다중) | 뽑을 관광 타입 한정 | 상수 |
| **동적** ☔ | **비 안 오는 곳만** | 지금 강수 없는 지역만 풀에 남김 | 기상청 초단기실황 API |
| **동적** 🎪 | **축제 진행 중인 곳만** | 오늘 진행 중 축제가 있는 지역만 남김 | TourAPI `searchFestival` |
| 정적 🌊 | **바다 보이는 곳만** | 뽑는 대상을 바다 관광지(해수욕장·섬 등)로 한정 | TourAPI 분류코드 상수 |
| **동적** 🦀 | **제철 재료 산지만** | 이번 달 제철 품목의 주산지 지역만 풀에 남김 | 자체 제철 달력 테이블 |
| **동적·프리컴퓨트** 🍃 | **한적한 곳만 (예측)** | 성수기 예측 시·도를 풀에서 제거 — 일 1회 배치 적재분 DB 조회 | 한국관광공사 관광지 집중률 예측 API (§6.7) |

- 걸린 조건들은 **AND**로 결합. 예: "강원+제주 중 · 비 안 오는 곳 · 맛집" 가능.
- **📅 방문 시점(§6.8)은 이 표의 조건이 아니다** — 걸린 동적 조건들의 판정 기준일만 바꾼다(날짜 단독 = 완전 랜덤).
- 동적 조건 상세 설계는 §6.

## 4. 데이터 흐름

```
[사용자] → [프론트엔드(Next.js)] → [API Route(서버)] ─→ [TourAPI]     관광지·축제
                                    ↑ 서비스키 숨김   └→ [기상청 API]  실시간 강수 (☔ 필터)
         ← ─ ─ ─ ─ ─  랜덤 여행지 1곳 (사진·주소·설명)  ─ ─ ─ ─ ←
```

### 왜 API Route(서버 중간층)가 필요한가 — 설계의 핵심

1. **서비스키 보호**: 브라우저에서 TourAPI를 직접 부르면 서비스키가 네트워크 탭·소스에 그대로 노출된다. 서버에서만 호출하면 키가 환경변수에 숨는다.
2. **CORS 회피**: 공공 API는 브라우저의 교차 출처 요청을 막는 경우가 많다. 서버-투-서버 호출은 CORS 제약이 없다.
3. **동적 필터 집계**: 여러 지역의 날씨를 병렬 조회·캐싱해 하나의 지역 풀로 합치는 로직(§6)은 서버에 있어야 깔끔하다.

→ Next.js의 **API Route(`app/api/random/route.ts`)** 하나로 이 중간층을 해결. 별도 백엔드 불필요.

## 5. TourAPI 활용

> 한국관광공사 국문 관광정보 서비스 — **KorService2 채택 확정**(2026-07 리서치·구현 검증).
> 베이스 `https://apis.data.go.kr/B551011/KorService2/` · **모든 오퍼레이션명에 접미사 `2` 필수**.

### 5.1 사용 엔드포인트 (KorService2)

| 엔드포인트 | 용도 |
|---|---|
| `areaBasedList2` | 지역·타입 기반 목록 조회 — **랜덤 추출의 메인** |
| `detailCommon2` | 개요(overview)·상세 보강 — **`contentId`만 넘긴다**(v2는 구버전 YN 플래그 제거, 넣으면 오류) |
| `detailImage2` | 추가 이미지 (선택) |
| `areaCode2` / `categoryCode2` | 지역·분류 코드 (초기 1회 또는 상수로 고정) |
| `searchFestival2` | 날짜 조건 축제 조회 — 🎪 **축제 필터의 소스** (오늘 진행 중 축제·지역 도출) |

### 5.2 공통 파라미터 & 응답 처리 (검증 확정)

- `serviceKey` (필수) · `_type=json` · `MobileOS=ETC` · `MobileApp=RandomTravel`
- `numOfRows` · `pageNo` (페이지네이션) · `arrange` (정렬, `O`=대표이미지 있는 것 우선) — M1은 arrange 미지정(균등성은 인덱스 난수화에서 나옴)
- **서비스키**: 반드시 **Decoding 키**를 `URLSearchParams.set`으로 넘긴다(정확히 1회 인코딩 → 이중 인코딩 `%252B`로 인한 키 미등록 오류 회피). Encoding 키·문자열 직결합 금지.
- **성공 판정**: HTTP 200이어도 실패일 수 있음 → 응답 `response.header.resultCode === "0000"`으로 판정. 인증·쿼터 오류(코드 30·22)는 게이트웨이가 **XML 봉투**로 주므로 JSON 파싱 실패 분기에서 잡아 안내(키 오류 vs 한도 초과 구분).
- **0건 방어**: 결과 0건이면 `body.items`가 빈 배열이 아니라 **빈 문자열 `""`**로 올 수 있음 → 배열 파싱 전 방어. 1건이면 `item`이 객체, 다건이면 배열로 옴(둘 다 흡수).

### 5.5 "진짜 랜덤" 알고리즘

TourAPI는 결과가 페이지로 쪼개져 오므로, 단순히 1페이지에서 고르면 항상 비슷한 결과가 나온다. 전체에서 균등하게 뽑으려면:

1. **전체 개수 확인** — 조건(지역·타입)으로 `numOfRows=1, pageNo=1` 호출 → 응답의 `totalCount` 획득
2. **랜덤 위치 선택** — `1 ~ totalCount` 중 정수 하나를 무작위 선택 (`randomIndex`)
3. **여행지 1건 조회** — `numOfRows=1, pageNo=randomIndex`로 해당 1건만 가져옴
4. **상세정보 결합** — 받은 `contentId`로 `detailCommon2` 호출해 개요·이미지 보강 → 결과 카드 완성

- **순수 랜덤 모드 (M1 구현)**: `contentTypeId`를 `RANDOM_DEFAULT_TYPES`(12·14·15·25·28)에서 **균등 선택**, `areaCode`는 미지정(전국) 후 위 알고리즘 적용 — **타입 균등 + 타입 내 항목 균등**(진짜 항목 균등 아님 — §13).
- **조건 랜덤 모드**: 사용자가 고른 `areaCode`·`contentTypeId`로 위 알고리즘 적용.

### 5.6 호출 예산·캐싱·중복 방지

"다시 뽑기"가 핵심 UX인데 뽑기 1회 = **API 2~3콜**(totalCount → 1건 → detailCommon)이라, 대비 없이는 개발계정 일일 쿼터(보통 1,000회 수준 — 활용신청 화면에서 실수치 확인)가 연타에 금방 닳는다.

- **totalCount 캐시**: 조합(지역×타입×cat3)별 totalCount는 거의 안 변함 → **24시간 캐시** → 재뽑기는 1~2콜로 감소.
- **detailCommon 지연 호출(선택)**: 카드 기본 정보(이름·주소·대표이미지)는 `areaBasedList` 응답만으로 구성 가능 — 개요(overview)는 카드 노출 후 로드하면 연타 비용 최소화.
- **중복 방지**: 뽑힌 `contentId`가 "최근 본 곳" 최근 N개(기본 10)에 있으면 재추첨(최대 2회). 단 **totalCount ≤ N이면 규칙 해제** — 작은 풀에서 무한 재뽑기 방지.
- **운영 전환**: 남에게 공유해 트래픽이 늘면 공공데이터포털 **운영계정 전환 신청**(쿼터 상향).
- **배치 예산(🍃 혼잡도, §6.7)**: 쿼터는 활용신청(서비스)별 **독립** — 일 1회 배치 ~265콜은 뽑기(KorService2) 쿼터를 잠식하지 않는다(잠식 0).

## 6. 동적 조건 설계 — 지역 풀 필터 파이프라인

모든 뽑기가 공유하는 파이프라인. 동적 조건은 2~3단계에 꽂히는 필터일 뿐이다.

```
지역 풀 (기본: 17개 시·도 전체 / 지역 조건 있으면 선택 지역만)
  → 🎪 축제 필터   (켜져 있으면: 진행 중 축제 없는 지역 제거)
  → ☔ 날씨 필터   (켜져 있으면: 지금 강수 있는 지역 제거)
  → 🦀 제철 필터   (켜져 있으면: 이번 달 제철 품목 주산지가 아닌 지역 제거)
  → 🍃 한적 필터   (켜져 있으면: 성수기 예측 지역 제거 — 배치 적재분 DB 조회, §6.7)
  → 남은 풀에서 지역 1개 랜덤 선택
  → §5.5 4단계 랜덤 추출 (테마 조건 반영 · 🌊 바다 조건 시 바다 분류코드로 한정)
```

요청 예: `/api/random?areas=32,39&types=39&noRain=1&festivalOnly=1&seaside=1&seasonal=1&quiet=1&date=20260712` — **파라미터를 전부 생략하면 완전 랜덤**. `date`는 풀을 좁히지 않는 **기준일 축**(§6.8) — 각 동적 필터의 판정 기준일은 기본 '오늘', 📅 칩으로 변경(☔만 오늘 전용).

### 6.1 ☔ 날씨 필터 — 기상청 초단기실황

- **소스**: 공공데이터포털 "기상청_단기예보 조회서비스"의 초단기실황(`getUltraSrtNcst`). TourAPI와 같은 포털이라 활용신청만 추가하면 됨.
- **판정**: 응답의 `PTY`(강수형태) 코드 사용. `0`(강수 없음)만 "비 안 옴"으로 취급.

| PTY | 의미 | PTY | 의미 |
|---|---|---|---|
| **0** | **강수 없음 ✅** | 3 | 눈 |
| 1 | 비 | 5 | 빗방울 |
| 2 | 비/눈 | 6·7 | 빗방울눈날림·눈날림 |

- **호출 방식**: 기상청 API는 격자 좌표(`nx`, `ny`) 기반 → **시·도별 대표 격자좌표**가 필요하다.
  - ✅ **M11 구현 결정**: 격자쌍 17개를 손으로 옮기면 오타 위험이 커서, `constants.ts`엔 시·도 대표 **위경도**(`AREA_LATLNG`)만 두고 격자 변환은 순수함수 `latLngToGrid`(기상청 LCC 공식, `lib/weather.ts`)로 계산한다. 서울(37.5665,126.978)→(60,127) 앵커를 단위 테스트로 못 박음.
- **비용 관리**: 풀에 남은 지역 수만큼 병렬 호출(최대 17). 초단기실황은 매시 40분경 발표 → **지역별 30분 캐시**(Next.js fetch `revalidate`)로 뽑기마다 재호출하지 않는다. ✅ base_time은 45분 컷으로 직전 정시 물러남(발표 지연 방어), 항상 KST.

- **품목-맛집 매칭(음식점 조합)**: 제철 필터를 **음식점(39)** 테마와 함께 걸면, 그 지역 랜덤 식당이 아니라 **제철 품목명으로 그 지역 맛집을 실제 검색**(`searchKeyword2`)해 재료-식당을 맞춘다. 이때 `SeasonalItem.dish`(회·해산물 true / 과일·채소 false)로 검색 대상을 가린다 — "수박 맛집"은 없으므로(농산물), 그 지역에 dish 제철거리가 없으면 일반 식당으로 **폴백하되 그 식당엔 제철 배지를 붙이지 않는다**('수박 제철 지역의 횟집' 같은 미스매치·오해 제거). dish 매칭 성공 시 배지는 **매칭된 그 품목만** 노출(지역 전체가 아니라 뽑힌 맛집과 일치). 로직 `lib/tourapi.ts`(`pickSeasonalRestaurant`)·순수부 `lib/season.ts`(`dishSeasonalItemsForArea`)·가드 `season.test.ts`.
- **축제 보강(선택)**: `searchFestival` 진행 중 축제명에 제철 품목명이 들어 있으면(예: 영덕대게축제) 카드에 함께 노출 — 정적 테이블에 '지금 살아있는' 신호를 더함.
- **시그니처 조합**: 제철 필터 + 테마 '음식점(39)' = **"지금 제철 산지의 맛집 랜덤"** — 위 품목-맛집 매칭으로 실제 이행(예: 7월 전남 → 완도 전복집 + 🐚전복 배지). UI 프리셋 버튼 후보.

### 6.5 빈 풀(조건 과다)·API 장애 대응

**빈 풀** — 필터를 다 통과한 지역이 0개일 수 있다(예: 장마철 전국 비).

- **2차 제안(택1)**: ① 조건 무시하고 그냥 뽑기 ② ☔ 상황이면 실내 테마(문화시설 14 등) 전환 제안

**API 장애·지연** — 공공 API는 응답이 수 초씩 걸리거나 간헐적으로 실패한다.

- **로딩 마스킹**: 뽑기 애니메이션(§3)이 도는 동안 API 호출 — 정상 응답이면 지연이 연출로 흡수된다. 애니메이션 **최소 노출시간**(예: 1초)도 보장 — ⏳ 설계만 있던 항목, **M18(§7.9)에서 구현**.

- **동적 필터 소스 장애**: ☔·🎪 API가 죽어 있으면 해당 필터만 건너뛰되, 건너뛴 사실을 결과에 표시(조용히 무시하지 않는다).
- **배치 데이터 stale(🍃 한적)**: 혼잡도 적재분이 낡았으면(조회 날짜 행들의 max(fetched_at) 48h 초과) 🍃 필터만 건너뛰고 표시, 후보 수 배지는 dynamic 폴백 — §6.7.

  - `export const maxDuration = 300` — **Fluid compute 활성 확인 필수**(레거시 함수 모드면 Hobby 상한 60s로 회귀).
  - 배치 절차: ① 집중률 250 시군구 × 1콜(동시성 8, 시군구별 실패 재시도 1회) — **응답 도착 즉시 집계 → 청크 upsert(≤1,000행)**: 한 방 INSERT는 바인드 파라미터 65,534 상한 임박이라 금지. 타임아웃·부분 실패에도 진행분 보존, 재실행만으로 자연 복구(idempotent) ② 방문자수 — 최근 가용일 역탐색(`visitorProbeDates`: 오늘-28일부터 과거로 최대 13일) 1일치 upsert, 집중률과 **병렬** ③ 보존(`retentionCutoff`): congestion은 `baseYmd < 오늘-7일` 삭제, visitor는 180일. **날짜를 가정하지 않는다** — 응답 행의 `baseYmd`를 그대로 키로 저장(KST 새벽엔 첫 행이 '오늘'이 아닐 수 있음).
  - **최초 적재**: cron은 **프로덕션 배포에서만 발화**(PR 프리뷰 미작동) → 로컬 dev 서버(`localhost:3456`, DB는 **공용 Neon**)에서 `/api/cron/congestion` 1회 호출이 유일한 사전 검증 경로. dev 모드 검증 생략이라 **시크릿이 셸 히스토리·채팅 어디에도 등장하지 않는다**.
  - 날짜 처리는 **항상 KST**(`kstYmd`·`visitorProbeDates`·`retentionCutoff` 전부 순수 함수 — 서버 UTC 함정, §6.1 base_time 계열).
- **⚠️ 코드 체계 함정 — 세 코드가 서로 다르다**: TourAPI areaCode(시·도 1~39) / 통계청 5자리(`KOREA_SIGUNGU.code`, 정복 지도) / **법정동 5자리(이 API들)**. 통계청·법정동은 **같은 값이 다른 지역을 가리킨다**(법정동 11110=종로구 vs 통계청 11110=노원구) — **코드값 직결 금지, (시·도 areaCode + 시군구명) 키 매핑**. `lib/congestionCodes.ts`(AUTO-GENERATED — `scripts/genCongestionCodes.mjs`가 `scripts/data/`의 **공식 코드표 xlsx**(활용 매뉴얼 동봉본, 출처 URL 주석)에서 생성)로 250개 전수 매핑 + **특례 4건**: ① 남구→미추홀구 28177(2018 개명) ② 세종시→세종특별자치시 36110 ③ **부천시→원미 41192+소사 41194+오정 41196 3코드 집계** ④ 군위군→대구 27720(2023 편입 — 앱 `area`는 경북 유지). 시·도 레벨 변환은 `LDONG_TO_AREA`(§6.2) 재사용.
- **데이터 흐름 — 판정·배지 공용 1회 조회**: `db/congestion.ts`(적재/조회 래퍼 — 스키마와 동거해 lib '순수 로직' 관례 보호, `lib/kma.ts`처럼 단위 테스트 비대상)의 `getCongestionDay(dateYmd)`가 **`baseYmd ≤ 기준일` 중 최신** 날짜의 전국 행을 일괄 반환. **`unstable_cache`(revalidate 3600)** 필수 — ① 일 1회 갱신 데이터 ② **neon-http 드라이버는 POST fetch라 기존 `revalidate` 캐시 관례가 안 먹히는 유일한 원격 소스** ③ Neon 무료 티어는 유휴 suspend(콜드 재개 0.5~2s)라 캐시 없으면 뽑기·후보 수(고빈도)에 직격.
  - **pctRank는 저장하지 않는다** — 조회 시점에 `rankDaily`(순수)로 그날 존재하는 전국 행 분포에서 계산. 배치 2차 패스 불필요 + 부분 실패로 잔존한 직전 성공 행도 분포에 자연 포함.
- **판정 (순수부 `lib/congestion.ts`, TDD — `QUIET_AREA_CUT` 등 상수도 여기 export)**:
  1. `aggregateDaily(rows)` — 시군구×날짜별 `cnctrRate` **중앙값**(`medianRate`) + 스팟 수 + 과밀 스팟 비율(`crowdedShare`, rate≥80 — 저장만, 후속 히트맵·판정 대안).
  2. `rankDaily(dayAggregates)` — 같은 날짜 전국 시군구 중앙값 분포의 **백분위 `pctRank`**(0=가장 한적). 고정 임계값이 아닌 분위수 — 전국 동시 계절 변동에 강건.
  3. `quietAreaCodes(ranked, cut)` — **시·도 판정: 소속 시군구 pctRank의 중앙값 ≤ `QUIET_AREA_CUT`(0.4)**. §6 시·도 풀 단위와 정합. 결측 시군구는 판정에서 제외(보수).
  4. `narrowByQuiet(base, quietSet)` — **순수부 동형 = `narrowByWeather`**(base: number[]|null + 집합 교집합). `drawRandom` 좁히기 **5단계**(🦀 제철 4단계 다음, 🌊 바다 분기 앞 — 기존 "5) 바다" 주석 번호는 6)으로 갱신). **래퍼 동형 = ☔ 날씨 단계**: try/catch로 `getCongestionDay` 조회 → 실패·stale 시 `notices.push`("혼잡도 데이터를 못 불러와 🍃 조건은 건너뛰었어요") 후 **필터·배지 모두 스킵**.
- **판정 기준일·신선도 (단일 규칙)**: 기준일 = `kstYmd(now)` — 📅 방문 시점 칩(**→ §6.8 M19로 정식 채택**)이 이 파라미터만 바꿔 그대로 연결. `getCongestionDay`가 반환한 날짜 행들의 **`max(fetched_at)` > 48h → stale** → 필터 소프트 스킵 + 후보 수는 dynamic 폴백. 30일 선행 예측이라 '오늘 이하 최신' 행은 배치가 하루 이틀 죽어도 존재 — **fetched_at이 실질 신선도 축**(cron 전달은 best-effort·재시도 없음·드물게 중복 — 중복은 upsert라 안전).
- **후보 수 배지 — 정적 편입(정확한 숫자)**: ☔·🎪(dynamic)과 달리 DB 조회라 결정적. **`planCandidateCount(p, month, quietSet?: Set<number> | null)` 시그니처 확장**(제철 calendar 주입과 동형 — 순수 유지). quiet 켜짐 + 교집합 0 → **`{ kind: "empty" }`** 반환. ⚠️ 라우트에서 지역 풀을 사전 교집합하는 방식은 금지 — CountParams의 "빈 배열=전국" 의미와 충돌해 **전멸이 전국으로 뒤집힌다**. quietSet 주입·stale 판단은 `countCandidates`(tourapi)에서 — stale은 예외가 아니라 **정상 경로로 `dynamic` 반환**(명시 분기).
- **결과 카드 근거 배지**: `PickedInfo.congestion?: { sigunguName: string; pctBelow: number; baseYmd: string }` — **`pctBelow = round(pctRank×100)`**. 계산은 `buildBadges` 동기 수행: **`BadgeCtx`에 `sigunguRanks: Map<법정동cd,{pctRank}>` + 기준 `baseYmd`를 실어**(weatherObs 동형) 뽑힌 좌표 → `sigunguAt`(§7.4 점-다각형, 최근접 스냅 ~7.5km) → (area+이름) → 법정동 → pctRank. 표기 예: **"🍃 집중률 하위 12% · 한적 예측 (7/10 기준)"**. **노출 조건: 🍃 켠 뽑기 + `pctRank ≤ 0.5`일 때만** — 필터는 시·도 단위라 개별 시군구가 높을 수 있어(시·도 중앙값 0.35에 시군구 0.68) 자기모순 카드 방지. 판정 실패·데이터 없음·0.5 초과 → 배지 생략(허위 서사 금지). `koreaMap` 서버 편입은 **클라 번들 무영향**(§7.4 지연 로드는 클라 원칙) — 콜드스타트 파스 비용만 소폭 수용.

### 6.8 📅 방문 시점 칩 (M19)

§11.1 백로그 "📅 방문 시점 칩"의 정식 채택. 뽑기의 **기준일**(오늘/내일/이번 주말(토)/다음 주말(토))을 선택해 **켜진 조건의 판정 날짜만** 바꾼다 — 집중률 API가 30일 예측이라 데이터는 이미 Neon에 있고(§6.7), `getCongestionDay`가 날짜별 캐시로 파라미터화돼 있어 **호출부의 날짜 인자만 바꾸면 되는** 저비용 확장. 진행 순서는 **M18(§7.9) 뒤** — 둘 다 `app/page.tsx`·FilterPanel을 만지고, 여기 신규 문구(☔ 잠금 안내·notice)가 §7.9 카피 원칙의 적용 대상.

- **불변식(최우선)**: 날짜는 필터가 아니라 **기준일 축** — **날짜 단독 선택은 후보 집합·뽑기 분포 무변**. 조건 0개면 `date`가 URL에 있어도 서버 동작은 완전 랜덤과 동일(§2 — 테스트 선작성). `hasAny`(= **'선택 초기화' 버튼·에러 패널 게이트** — 후보 배지는 hasAny 무관 상시 표시)에 date 불포함 유지하되, **초기화 버튼 노출 조건은 `hasAny || dateYmd !== null`로 확장**('내일'만 선택 시 초기화 대상은 있는데 버튼이 없는 UI 모순 방지). `clearFilters`엔 포함('오늘' 복귀). **상시 마이크로카피 1줄**(날짜 행 바로 아래): "날짜는 조건을 판정할 기준일이에요 — 날짜만으론 후보가 줄지 않아요."(칩은 하이라이트인데 하단 '아무것도 안 고르면 완전 랜덤' 문구가 뜨는 시각 모순 해소 + "날짜가 필터인가요?" 오해 선제 답변).
- **요청 파라미터**: **`date=YYYYMMDD`**(절대 날짜 — 주말 계산은 클라 몫, 서버는 검증만). `lib/query.ts`에 **`parseDateYmd(raw, now)`** 신설: 8자리·실존 날짜 + 범위 **`오늘 ≤ date ≤ 오늘+29`(KST — 집중률 예측 창)** 밖이면 null(무시 — 경계 정리 관례). '오늘' 선택 시 클라는 **파라미터 자체를 생략**. **요청당 단일 시계**: 라우트에서 `const now = new Date()` 1회 생성 → `parseDateYmd(raw, now)`와 `drawRandom({ …, dateYmd, now })`에 동일 주입(자정 걸침 시 검증 기준일·판정 기준일 어긋남 방지).
- **날짜 파생 규칙** — `DrawParams.dateYmd?: string` 신설. **우선순위 명문화: 명시 주입(`month`·`today`) > `dateYmd` 파생 > 현재시각**(테스트·호출부가 동시 주입해도 결정적):
  - 🦀 제철: `month = params.month ?? (dateYmd ? monthOf(dateYmd) : currentMonth())` — 월말에 다음 달 주말 케이스 포함.
  - 🎪 축제: `getFestivalMap(params.today ?? params.dateYmd ?? todayKST())` — `eventStartDate`가 URL에 들어가 캐시 자동 분리, `isInProgress` 판정 그대로.
  - 🍃 한적: `getCongestionDay(params.dateYmd ?? kstYmd(params.now))` — §6.7 확장 훅 그대로. stale 판정 `nowMs`는 실제 현재 시각 유지.
  - 상충 주입(예: `month`와 `dateYmd` 동시) 케이스 테스트 1개 추가.
  - ☔ 날씨: **미래 불가(구조적 — 초단기실황은 현재 관측만)**. **전이 규칙(🌊-types 패턴 그대로)**: 오늘 아닌 칩 선택 시 ☔ 토글 **잠금 표시**(잠긴 토글은 눌리지 않은 것으로 취급 — 🌊 관례) + **`buildRandomQuery`가 date≠오늘이면 `noRain` 미방출**(표시·전송·서버 일치, 선택 state 보존 — 오늘 복귀 시 복원; count 경로도 noRain 미방출이라 dynamic 강등 없음). **서버 무시는 `dateYmd > 오늘(KST)`일 때만**(직접 URL `date=오늘&noRain=1`은 범위상 유효 — 죽이지 않음) + notice "오늘이 아닌 날짜라 ☔ 조건은 건너뛰었어요."(직접 URL 방어 전용 — 정상 UI 경로에선 안 뜸). 잠금 안내 문구: "☔ '비 안 오는 곳'은 지금 날씨만 알 수 있어요 — 그래서 오늘 뽑기에서만 켤 수 있어요."(🌊 이유 설명형 문형). 단기예보(`getVilageFcst`, +3일 POP) 확장은 §11.1 백로그.
- **count 경로**: `countCandidates`를 **옵션 객체로 개편** — `countCandidates(params, opts?: { month?: number; now?: Date; dateYmd?: string })`(현행 2번째 위치 인자가 `month: number`라 dateYmd를 그 자리에 못 넣음; 직접 단위테스트가 없어 시그니처 변경 부담 낮음). 내부 파생은 위와 동일 규칙 — `month = opts.month ?? (dateYmd ? monthOf(dateYmd) : currentMonth())`, quietSet은 `getCongestionDay(dateYmd ?? kstYmd(now))`. 🍃 정적 편입·empty·dynamic 분기는 §6.7 그대로.
- **주말 계산**(신규 순수부 `lib/tripDate.ts`, TDD): `dateChips(now)` → `[{key, label, ymd}]`. **정의(요일 기준 명시): 오늘이 토·일이면 '이번 주말' = 오늘**(→ ymd 중복으로 자연 생략), **그 외엔 다가오는 토요일. '다음 주말' = 그다음 토요일**(일요일에 '다가오는 토요일'=6일 뒤가 '다음 주말'과 충돌하는 파탄 방지). 중복 ymd 칩 생략(개수 가변 3~4개). **칩 라벨 "이번 주말(토)"·"다음 주말(토)"** — '주말'(토·일 이틀)을 약속하면 일요일 진행 축제 누락 반례에 무방비라 토요일 기준을 정직 표기. 토·일 합집합 판정은 §11.1 백로그로 명시 분리. KST 헬퍼는 **신설 `lib/kst.ts`**로 이동(`kstDateParts`·`ymdOffset`·`kstYmd` — `lib/congestion.ts`는 re-export로 기존 소비처 무변): congestion을 직접 import하면 클라 번들에 congestionCodes 대형 테이블 유입 + 도메인 모듈에 범용 헬퍼가 사는 계층 역전. 테스트: 요일별 7케이스(**일요일 기대값 명시**) + 월말·연말 경계 + 중복 생략.
- **UI(FilterPanel)**: 지역·테마와 같은 **가로 pill 행**(`chip()` 재사용, 단일 선택 라디오형) — 테마 아래·추가 조건 위 "📅 언제 가요?" 섹션, 기본 '오늘' 상시 노출(숨김 확장이면 데모에서 기능이 안 보임 — 노출형 유지). **선택 상태는 ymd로 저장** — 렌더마다 `dateChips(now)`와 조인해 일치 칩 하이라이트, **일치 칩이 없으면(자정 통과·과거화) '오늘' 자동 복귀**(state 초기화 — key 저장 시 고른 날짜가 소리 없이 바뀌는 정직성 위반 방지). 자정 경계 테스트: 토 00:00 직후 금요일 선택 ymd가 어느 칩에도 안 붙는 케이스.
- **결과 카드 — 배지 문구 개정 포함**(기존 "(M/D 기준)"의 '기준'이 '데이터 기준일'로 오독되는 양의성 + 배치 지연 시 `baseYmd < 선택일`이면 고른 날짜와 다른 날짜가 떠 버그처럼 보임): 🍃 배지 = **"🍃 {M/D} 한적 예측 · 집중률 하위 N%"**(예측 대상일 선두 — 정상 경로에선 선택일과 일치), **`baseYmd < 선택일` 예외에만 "({M/D} 데이터)" 병기**. 🎪 축제 배지 = **오늘 아닌 기준일이면 "(M/D 기준)" 부가**(시작 전 축제가 '진행 중'처럼 보이는 오해 방지 — §7.9 원칙 2 기준일 명시).
- **테스트**: 신규 — `kst.test.ts`(이동분) · `tripDate.test.ts`(위) · `query.test.ts`에 `parseDateYmd`(형식·실존·범위 오늘/+29 경계) + `buildRandomQuery`(date 방출/오늘 생략/**date≠오늘 시 noRain 미방출**) · "date 단독 = 계획 무변"(`candidateCount.test.ts`) · 날짜 파생 우선순위·상충 주입. 갱신 — `candidateCount.test.ts`(month 파생). 축제·한적 판정 자체는 기존 테스트 커버(날짜는 통과 인자).

  - ⚠️ **2026-07-21 무료 쿼터 정책 변경 예고**(계정당 첫 활성화 앱 1개 한정 등) — 구현 시점에 최신 공지 확인.
- **키 취급**: JS SDK 키는 브라우저 노출이 전제인 키(도메인 제한으로 보호) — TourAPI 서비스키와 달리 `NEXT_PUBLIC_` 허용.
- **시너지**: §12 수익 모델의 방문 인증(`visited`) 이벤트와 정확히 맞물림 — 지도에 핀이 쌓이는 재미가 방문 체크의 동기가 되어, P1 로깅 도입 시 이 체크가 곧 `visited` 이벤트가 된다.

### 7.2 딥링크 저장 유도 (반자동)

결과 카드의 "지도 앱에서 열기" 버튼 → 해당 앱에서 열리면 사용자가 직접 ★저장 한 번 탭. 3사 모두 공식 경로 확인됨:

| 앱 | 링크 | 비고 |
|---|---|---|
| 카카오맵 | `kakaomap://place?id={장소ID}` / 웹 `map.kakao.com/link/map/{id}` | 공식 URL 스킴 문서 존재 |
| 네이버지도 | `nmap://place?lat=..&lng=..&name=..&appname={식별자}` | `appname` 필수 · 미설치 시 스토어 유도 |
| 구글맵 | `https://www.google.com/maps/search/?api=1&query=위도,경도` | API 키 불필요 · 앱/웹 자동 분기 |

- TourAPI 응답의 좌표(`mapx`/`mapy`)·장소명으로 링크 생성. 카카오 장소ID는 TourAPI와 체계가 달라 **좌표 기반 링크가 기본**.
- **PC/모바일 분기**: `kakaomap://`·`nmap://` 앱 스킴은 PC 브라우저에서 무반응 → 앱 스킴은 **모바일에서만**, PC에선 웹 URL(카카오 `map.kakao.com/link/…`, 네이버 지도 웹 검색 URL)로 폴백. 구글맵 URLs는 자체 자동 분기라 그대로 사용.

### 7.3 KML 내보내기 (백로그)

방문 목록 → KML 파일 다운로드 → 사용자가 **구글 My Maps에 수동 임포트**(공식 지원, 레이어당 2,000행 제한). 임포트 자동화 API는 없음 — 파워유저용이라 수요 확인 후 구현.

### 7.4 🧩 전국 정복 지도 (M12, 정식 기능)

- **동작**: `/map`에 **[📍 핀 지도 ↔ 🧩 정복 지도]** 토글 추가. 정복 지도는 대한민국 **250개 시·군·구**를 실제 윤곽 SVG 조각으로 그리고, 다녀온 곳을 **1곳 이상 방문한 시·군·구를 색(indigo)으로 채운다**. 안 가본 곳은 회색 빈 조각. 방향 잡기용 **시·도 굵은 외곽선** 오버레이. 상단 **"N / 250 · xx%" 게이지**, 하단 **시·도별 진행 칩**("서울 3/25" 식).
  - **단위 = 시·군·구 이유**: 시·도(17)는 너무 굵어 17번이면 100%라 "완주"의 무게가 안 산다. 시·군·구(약 250)면 완주 바가 유의미해진다(2026-07-05 사용자 피드백).
- **정복 판정 = 좌표 점-다각형**: 방문 데이터엔 시·군·구 코드가 없다(시·도 `areaCode`만). 그래서 방문 **좌표(lat/lng)를 지도 평면으로 투영**한 뒤 **어느 시·군·구 다각형 안에 있는지 even-odd ray casting**으로 판정한다. → **스키마 변경 0, 기존 방문 기록도 소급 반영**. 좌표 없는 방문만 제외(대부분 TourAPI 좌표 있음). **추가 API·키 불필요**(카카오 키 없이도 동작, 순수 SVG).
- **핵심 설계**: 윤곽은 손으로 못 옮기는 산출물이라 M11과 같은 원칙 — **생성기로 재현**(`scripts/genKoreaMap.mjs` → `lib/koreaMap.ts`). 통계청 파생 시·군·구/시·도 경계 GeoJSON(southkorea-maps, public domain)을 받아 작은 섬 제거 + Douglas-Peucker 단순화 + 위도 보정 등거리 투영. 시·군·구·시·도를 **같은 투영**으로 겹치게 방출하고, 판정용 **투영 파라미터(`KOREA_PROJECTION`)**도 함께 내보내 런타임 판정과 렌더가 같은 평면을 쓰게 한다. ⚠️ 통계청 code(앞2자리)와 **TourAPI areaCode는 35/36/37/38이 뒤바뀌어** 있어 `PREFIX_TO_AREA`로 매핑.
- **순수 로직**: `lib/conquer.ts`(TDD) — `projectLatLng`·`pointInRings`(even-odd)·`sigunguAt`(bbox 프리필터)·`conqueredSigunguCodes`·`conquerStats`(개수/250/퍼센트)·`conquerByArea`(시·도별 진행). 렌더는 `components/ConquerMap.tsx` — 시·군·구 데이터(약 200KB)라 **정복 뷰 전환 시에만 `next/dynamic` 지연 로드**(핀 뷰 기본 로드는 가볍게).
- **경계 갭·해안 스냅 폴백(2026-07-05)**: 두 가지 무성 미스를 함께 구제한다. ① 인접 시·군·구 경계를 각자 독립 단순화(DP)해 생기는 서브픽셀 갭. ② 해안선 단순화(EPS ~450m)·소도서 제거가 **해안·반도 관광지 좌표를 폴리곤 밖으로** 밀어냄(실측: 인천 월미도 관광지가 중구 경계 밖 0.04px → 첫 릴리스에서 정복이 0으로 샜음, 사용자 제보). 관광지 좌표는 늘 육지·해안이라, 정확 판정 실패 시 **가장 가까운 시·군·구 경계까지 거리 ≤ 12px(~7.5km)면 그 조각으로 스냅**한다(초기 8방향 nudge는 뾰족한 돌출부를 못 잡아 교체). 실측상 해안 미스 ~0.04px, 먼바다(동해 한가운데) ~103px라 그 사이 임계로 해안·소도서는 구제하되 먼바다는 배제. 회귀 테스트로 월미도 좌표를 못 박음.
- **알려진 한계**: `KEEP_RATIO`로 잘린 부속 도서(예: **독도**)는 투영 bbox 밖이라 판정 불가 — 넣으면 지도가 동쪽으로 크게 늘어나 구성이 망가져 **의도적으로 감수**(독도는 사실상 랜덤여행 POI 아님). 근본 해결(topojson 위상 보존)은 현 범위 밖.
- **시너지**: 백로그 "🏴 전국 정복률"의 시각화 승격. 조각이 채워지는 재미가 **"다녀왔어요 ✔" 체크 동기**를 강화 — §12 `visited` 이벤트와 맞물림(§7.1과 동일 취지).

### 7.5 💬 뽑기 결과 카톡 공유 (M13)

- **동작**: 결과 카드에 "💬 카카오톡 공유" 버튼. 누르면 카카오 JS SDK `Kakao.Share.sendDefault`(feed 기본 템플릿)로 **채팅에 사진+제목+주소 카드**가 꽂힌다. 카드/"지도에서 보기" 버튼은 그 장소의 **카카오맵 링크**(§7.2 `lib/mapLink` 재사용), "나도 뽑기" 버튼은 앱 홈. **새 라우트 없이** 구현(사용자 결정).
- **왜 시·도 SDK와 별개**: 지도 SDK(`dapi.kakao.com`, `window.kakao.maps`)와 카카오 JS SDK(`t1.kakaocdn.net/.../kakao.min.js`, `window.Kakao`)는 **별개 스크립트·별개 전역**이라 공존한다. 공유는 후자 필요 → 첫 클릭에 1회 지연 로드(SRI integrity 검증) + `Kakao.init(NEXT_PUBLIC_KAKAO_MAP_KEY)`. **같은 JavaScript 키 재사용**, 콘솔 [제품 설정 > 카카오톡 공유]만 활성이면 됨(sendDefault 기본 템플릿이라 메시지 템플릿 등록 불필요).
- **폴백**: SDK 미로드·비카톡 환경 → `navigator.share`(모바일 공유시트, 카톡 포함) → 실패 시 클립보드 복사("링크 복사됐어요" 인라인 안내). 토스트 시스템은 안 만듦.
- **이미지 https**: 카톡 피드 `imageUrl`은 https 필수 → TourAPI 이미지를 http→https 업그레이드(`httpsImage`), 이미지 없으면 앱 아이콘(`/icon-512.png`)로 대체.
- **순수 로직**: `lib/kakaoShare.ts`(TDD) — `buildShareFeed`·`httpsImage`·`shareText`. SDK 배선은 `hooks/useKakaoShare`(useKakaoLoader 싱글턴 패턴 재사용), 렌더는 `components/ResultCard`.
- **시너지**: §11.1 백로그 "결과 공유 URL"의 **선행 구현**(무료 유입 경로). 붙여넣기 링크 언펄링용 `/place/[contentId]`+OG 이미지는 잔여 백로그.

### 7.6 📍 첫 여행지 주변에서 뽑기 (M14)

- **동작**: 순수 랜덤으로 첫 여행지를 받으면 그 결과가 **앵커(여행 거점)** 가 된다. 결과 카드에 **"📍 {첫 여행지} 주변에서 뽑기"** 버튼이 뜨고, 누르면 앵커 좌표 **반경 20km** 안에서 랜덤 1건을 추천한다. "다시 뽑기"(전국 랜덤)는 그대로 — 주변 뽑기는 **별도 버튼**(사용자 결정). 주변 결과 카드엔 **"📍 {앵커}에서 N km"** 거리 배지.
- **앵커 규칙**: 앵커 = 가장 최근 **전국 랜덤 결과**(뽑기/다시뽑기 성공 시 갱신). 주변 뽑기는 앵커를 **바꾸지 않아**, 반복해 눌러도 같은 거점 주변을 계속 탐색한다.
- **범위**: **순수 랜덤 모드 전용**(사용자 결정) — 조건 랜덤(지역·테마·바다·제철·축제·날씨)은 기존 동작 유지, 주변 버튼 미표시. 첫 여행지에 좌표가 없으면 버튼 숨김(폴백 = 그냥 다시 뽑기).
- **서버**: TourAPI **`locationBasedList2`**(요청 `mapX`=경도·`mapY`=위도·`radius`·`contentTypeId`·`arrange="E"`, 응답 `dist`(m)). 기존 `tourFetch`·`pickItemFrom` 재사용, `getTotalCount`/`getItemAt`를 `(endpoint, params)`로 일반화. `RANDOM_DEFAULT_TYPES` 셔플→첫 비어있지 않은 타입에서 랜덤 인덱스 1건(타입 균등). 반경 내 0건이면 `EMPTY_POOL` 안내. **추가 API·키 불필요**(같은 `TOUR_API_KEY`).
- **순수 로직**: `lib/query.ts`(`parseLatLng`·`buildNearbyQuery`, TDD)·`lib/geo.ts`(`formatKm`, TDD). API `?near=위도,경도` → `drawNearby`. 클라이언트 `app/page.tsx`(anchor state), 렌더 `components/ResultCard.tsx`.
- **기록 서랍에서도 진입(후속)**: 결과 카드는 새로고침·시간 경과로 사라지므로, 서랍(찜·최근·다녀옴)의 **좌표 있는** 각 기록 행에도 **📍 버튼**을 달아 그 장소를 앵커로 삼아 주변 뽑기를 걸 수 있게 했다(`components/RecordPanel.tsx`). 이때 앵커는 그 기록으로 바뀐다. 기록에서 주변 뽑기를 하면 조건 모드여도 **거리 결과 카드**엔 이어서 탐색하도록 주변 버튼을 노출(카드 기본 노출은 여전히 순수 모드 전용).

### 7.7 📊 재방문 의향 평가 (M15)

- **동작**: **다녀옴** 탭의 각 기록 행 둘째 줄에 **3단계 재방문 의향** 세그먼트(가로 3분할)를 단다 — **👎 또 안 감**(또 갈 생각 없음) · **😐 그럭저럭**(나쁘지 않았음) · **👍 또 갈래**(또 갈 의향이 있음). 별점의 애매함 대신 "또 갈까?"라는 실제 의사결정 축. 선택된 칸만 색(rose/amber/emerald)으로 채우고, 같은 칸을 재클릭하면 **해제**(미평가). 전체 문구는 `aria-label`.
- **범위·위치(사용자 결정)**: **다녀옴 탭 전용**(찜·최근엔 미표시, 결과 카드도 미표시 — 카드는 깔끔 유지). 좌표 유무와 무관(평가는 반경 검색과 별개).
- **저장·동기화**: 평가는 **방문 장소의 속성**(`SavedPlace.rating?: 1|2|3|null`). localStorage `rt.visited.v1` 항목에 함께 실리고(직렬화 왕복 보존), 로그인 시 **write-through**(POST 업서트가 `rating`까지 갱신)로 기기 간 동기화. DB `user_place`에 nullable `rating` 컬럼 추가(additive — 기존 코드 호환). 병합 충돌 시 `rating`은 **null 아닌 쪽 보존**(coalesce)해 평가 유실 방지.
- **순수 로직(TDD)**: `lib/travelStore.ts`(`setRatingInList`)·`lib/placesApi.ts`(`sanitizePlace`가 1\|2\|3만 통과)·`lib/syncMerge.ts`(rating coalesce). 라벨 상수 `REVISIT_OPTIONS`(`lib/constants.ts`). 클라이언트 `hooks/useTravelStore.ts`(`setRating` write-through), 렌더 `components/RecordPanel.tsx`.
- **v1 한계**: 비로그인 상태로 평가 후 로그인 시, 서버에 이미 있는 장소면 sync 업로드(`onConflictDoNothing`)가 그 평가를 안 올릴 수 있음(기존 "삭제 미전파"와 동류). write-through가 주 경로라 실사용 영향 작음. **후속 후보**: 평가 기반 필터·주변 뽑기 제외·지도 마커 색·rating 이벤트 로깅(§12).

> ⚠️ **판매·서버 수집은 P0(MVP)에 반영하지 않는다.** 사용자 규모가 생긴 뒤를 전제로 한 확장 설계. 단, 이벤트 스키마(§12.5)는 **P0부터 localStorage에만 기록**해 미리 검증한다(§12.6 P0).

### 12.1 컨셉

사용자가 **뽑고 → 반응하고(찜/다시 뽑기) → 실제로 다녀오는** 과정에서 쌓이는 행동 데이터를 **익명 통계·트렌드 리포트**로 가공해 관광 업계(지자체·여행사 등)에 판매한다. 통신사·카드사가 지역 관광 소비 데이터를 지자체에 파는 것과 같은 구조의 축소판.

**이 데이터만의 차별점**: 일반 여행 플랫폼 데이터는 "검색·예약"이라는 *계획된 의도*지만, 이 서비스의 데이터는 **"무작위로 던졌을 때 사람들이 무엇을 수락하고 무엇을 거절하는가"** — 편향이 적은 순수 선호도 신호에 가깝다.

### 12.2 무엇이 데이터 자산이 되나

| 데이터 | 내용 | 팔리는 이유 |
|---|---|---|
| 뽑기 조건 로그 | 어떤 지역·테마 조합이 선택되는가 | 여행 수요 **선행지표** |
| 결과 반응 | 다시 뽑기(거절) vs 찜·길찾기(수락) | 여행지 단위 선호도·매력도 점수 |
| 방문 인증 (확장 기능) | 추천 후 실제 방문 여부 | 추천→방문 **전환율**, 실수요 증명 |
| 시간 패턴 | 계절·요일·시간대별 뽑기 경향 | 수요 예측, 캠페인 타이밍 |

### 12.3 잠재 구매처

| 구매처 | 사고 싶어할 것 |
|---|---|
| 지자체·지역관광공사 | 우리 지역 관심도·거절률, 어떤 테마로 소비되는지 |
| 여행사·OTA·숙박 플랫폼 | 뜨는 지역 조기 감지, 상품 기획 근거 |
| 상권분석·컨설팅 업체 | 관광 수요 보조 데이터 |
| 광고주 | (데이터 직판 대신) 지역·테마 타겟 광고 지면 |

### 12.4 ⚖️ 법적 전제 — 이 모델의 성립 조건

개인을 식별할 수 있는 데이터를 동의 없이 제3자에게 판매하는 것은 **개인정보보호법 위반**이다. 합법적으로 성립하는 형태는 셋:

| 형태 | 판매 | 비고 |
|---|---|---|
| **익명정보 (집계 통계)** | ✅ 자유 | "6월엔 제주×맛집 조합이 가장 많이 뽑혔다" 수준의 집계. **← 이 프로젝트가 채택하는 형태** |
| 가명정보 | ⚠️ 제한적 | 데이터 3법 이후 통계·연구 목적 활용 가능하나 절차 요건 있음 |
| 동의 기반 제3자 제공 | ⚠️ 가능은 함 | 명시적 동의 필요 — 사용자 거부감 크고 동의율 낮음 |

- **개인 단위 raw 데이터는 팔지 않는다.** 상품은 항상 집계·통계 형태로만.
- 방문 인증에 GPS 자동 수집을 쓰면 **위치정보법**(위치정보사업 신고 등)이 추가로 걸린다 → 방문 인증은 **사용자 수동 체크** 방식으로 설계해 회피.
- 이벤트 수집을 시작하는 시점부터 **개인정보처리방침 고지**가 필요하다.

### 12.5 아키텍처 영향 — 현행 설계와의 충돌

현행 "DB 없음·로그인 없음·localStorage" 전제를 데이터 사업화가 깬다. 다만 **로그인은 계속 없어도 된다** — 익명 세션 ID로 충분.

| 현행 (P0) | 데이터 사업화 시 필요 |
|---|---|
| localStorage에만 기록 | 서버 이벤트 수집 API + DB |
| 계정 없음 | 계정 불필요 유지 — 익명 세션 ID(UUID)면 충분 |
| 개인정보 미수집 | 식별자 최소화로 익명 유지 + 처리방침 고지 |

이벤트 스키마 초안:

```jsonc
{
  "event": "draw | redraw | like | navigate | visited",
  "mode": "pure | filtered",   // draw·redraw 에만 유효. like·navigate·visited 는 뽑기 시점 모드를 알 수 없어 null (M4 구현 확정)
  "areaCode": 39,
  "contentTypeId": 39,
  "contentId": "뽑힌 여행지 ID",
  "sessionId": "익명 UUID — 기기·개인 식별 불가",
  "ts": "발생 시각"
}
```

### 12.6 단계 로드맵

| 단계 | 내용 | 착수 조건 |
|---|---|---|
| **P0** | 수익화 없음. 단 §12.5 스키마로 이벤트를 **localStorage에만 기록** — 서버 전송 없음 = 법적 이슈·처리방침 불필요. P1 전환 시 "내 기록 업로드" 옵션 + 스키마는 이미 실전 검증된 상태 | 지금 (2026-07-03 확정) |
| **P1** | 익명 이벤트 로깅 추가 (위 스키마) | 배포하고 남에게 공유하기 시작할 때 |
| **P2** | 월간 트렌드 리포트 자동 생성 — 판매 샘플이자 홍보 콘텐츠 | 이벤트 수천 건 누적 |
| **P3** | 구매처 접촉·데이터 상품화 | MAU 수천 이상 + 사업자등록·법률 검토 |

### 12.7 현실 체크

- **데이터는 규모가 있어야 상품이 된다.** 혼자 쓰는 단계에선 가치가 0이므로 P0에 서버 수집은 넣지 않는다. 대신 로컬 기록(§12.6 P0)으로 스키마를 미리 돌려두고, P1 로깅을 일찍 켤수록 누적 히스토리 자체가 자산이 된다.
- 같은 노력 대비 진입장벽이 낮은 대안 수익원: 숙박·액티비티 **제휴 링크(어필리에이트)**, 광고. 데이터 판매는 이들보다 단가는 높지만 훨씬 뒤 단계의 이야기다.
